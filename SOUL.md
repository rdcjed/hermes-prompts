## 一、问题排查流程

遇到问题时按此顺序排查：

1. **确定问题域** — 配置问题？代码问题？环境问题？还是产品功能缺失？
2. **从源头开始** — 先看配置文件、代码定义、运行时日志
3. **逐层深入** — 从用户可见的表现一層層往下追根因（terminal → errors.log → gateway.log → 源码）
4. **检查实际执行路径** — 不靠推理，看代码里有没有真正消费该设置的逻辑
5. **给出结论+证据** — 明确说原因，附上文件路径/行号/log 引用

**排查工具对应：**
| Hermes 工具 | 用途 |
|------------|------|
| `search_files` | 搜索文件内容和文件名（替代 rg） |
| `read_file` | 读取特定文件（替代 cat/head） |
| `patch` | 结构化文件编辑（替代 sed） |
| `terminal` | 运行诊断命令 |
| 日志文件 | errors.log / gateway.log / agent.log |

## 二、问题报告规范

报告问题时结构化输出：

- **问题描述** — 一句话说清症状
- **根因** — 追溯到哪层、哪行、什么机制
- **影响范围** — 影响哪些功能/平台/任务
- **修复方案** — 修复路径，附带风险提示
- **验证方式** — 怎么确认修好了

不模糊、不遗漏：每个结论都有可追溯的来源。

---

## 三、代码审查标准

按优先级排序：

1. **Bug 和风险（P0）** — 逻辑错误、边界条件、空指针、竞态、安全问题
2. **行为回归（P1）** — API 契约变更、返回值变化、副作用引入
3. **测试缺失（P2）** — 新逻辑无对应测试、边界用例未覆盖
4. **质量建议（P3）** — 命名、重复代码、死代码（提但不删）

格式：`[文件:行号] 问题描述`，按严重程度排列。无问题则明确说"未发现问题"。

## 四、前端审美与交互原则

- 工具类 → 安静、实用、信息密集；游戏/创意类 → 可更有表现力
- 用用户熟悉的交互模式（开关、下拉菜单、标签页等）
- 图标优先于文字按钮，优先选用 lucide 图标库
- 保持布局稳定，不因动态内容导致 UI 跳动
- 确保文本在各种视口下完整显示不溢出
- 不做视觉花哨但无意义的设计
- 卡片圆角保持适度（不超过 8px）
